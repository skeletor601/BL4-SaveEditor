Item Editor HTML vs universal_parts_db.json
============================================

Extraction (reference htmls/Borderlands Item Editor and Save Editor.html):
  Rows extracted to CSV:     5092  (item_editor_parts.csv)
  Unique part codes:         5092

Our DB (universal_parts_db.json):
  Total rows:                9621
  Unique part codes:         9621  (no duplicate codes)

Overlap:
  In both (by code):         5071
  Only in Item Editor:         21  (we are missing these in DB)
  Only in DB:                4550  (from other sources: BL4 Master List CSVs, sources/*.csv, etc.)

Why we have 9000+ rows (and what's wrong):
------------------------------------------
1. We merge multiple sources. build_universal_parts_db.py pulls in:
   - master_search/db/sources/*.csv and *.json (e.g. bl4_parts_database.csv, parts_database_canon_v2_split_columns.csv)
   - master_search/db/Borderlands 4 Item Parts Master List - *.csv (Enhancements, Grenades, Repkits, Shields, Weapon Elemental, Heavy Weapons, etc.)
   So we get ~5k from Item Editor (canonical game data) plus ~4.5k from those other CSVs.

2. All 4550 "only in DB" rows have EMPTY Part Type and EMPTY String. They are stub rows: a code (or ID) was imported but no part type or name was filled. So they add row count but little value for search/display.

3. Many of those 4550 codes look malformed (e.g. {16:16:72}, {256:s457})—likely from CSV columns that were parsed as "prefix:part" incorrectly (e.g. "16:72" becoming prefix 16 and part "16:72", or string IDs like "s457" in the part field).

Recommendations:
- Treat Item Editor as canonical (~5k parts). Optionally prune DB to only rows that have Part Type or String filled, which would drop the 4550 stubs and bring total near 5k–6k.
- Or: keep merging other sources but backfill Part Type / String from Item Editor when code matches, and fix code formatting when importing from CSVs so we don't create {prefix:prefix:part} or {x:sy} styles.
